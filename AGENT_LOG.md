## Part 1 — Development Process & Coding Agent Usage

### Tool used

**GitHub Copilot** (agent mode, Claude Sonnet 4.5 via VS Code) was the primary coding agent throughout. All non-trivial code generation, scaffolding, and debugging was done through Copilot's agent interface — not just single-line completions.

---

### What the agent generated

#### Project scaffolding & architecture
I defined the architecture upfront: Feature-Sliced Design (FSD) layout (`shared → entities → features → widgets → app`) for the frontend, and the NestJS module split (`auth`, `boards`, `cards`, `columns`, `labels`, `users`, `workspaces`, `realtime`, `ai`) for the backend. Once the structure was set, I handed it to Copilot as a constraint and asked it to scaffold files within that layout — it didn't propose the structure, it implemented what I specified.

#### Backend modules (NestJS)
Each NestJS module was generated with a single prompt describing the entity, its relationships, and the required endpoints. For example, the `boards` module was generated by prompting:
> "Generate a NestJS module for boards. A board belongs to a workspace. It has a name, description, and a list of columns. JWT-protected CRUD endpoints. Use Mongoose."

Copilot produced the schema, DTOs, service, and controller. I reviewed for correctness and adjusted:
- Added missing `@Prop({ default: [] })` for embedded arrays (Copilot sometimes omitted defaults)
- Fixed CORS configuration (generated config used `origin: '*'`, changed to env-controlled allow-list)
- Added the `revalidatePath` calls in server actions (Copilot missed Next.js cache invalidation in first pass)

#### Frontend components & pages
UI was scaffolded almost entirely by Copilot:
- **`card-detail-modal.tsx`**: prompted as "3-column modal — description editor, metadata sidebar, AI chat panel". Copilot produced the layout; I iterated on the exact column widths and the AI chat panel integration.
- **`board-view.tsx`** with `@hello-pangea/dnd`: generated the DnD wiring; I fixed the column reorder logic (Copilot's first version only handled card moves within the same column).
- **`board-header.tsx`**: AI Assist dropdown and side panels generated by Copilot; I lifted `prioritizeResult`/`standupResult` state up to survive panel open/close (Copilot initially put state inside the Sheet content which unmounts on close).
- **Sidebar, workspace pages, auth forms**: generated wholesale with minor styling tweaks.

#### AI agents (the product AI layer)
Copilot generated the initial version of all four agents in `ai.service.ts` from a single large prompt describing each agent's behavior. Significant manual work was needed here:
- **Prioritizer**: Copilot's first version sent cards directly to GPT without any pre-scoring. I added the numeric urgency formula (`priority_weight + age_days/7 + overdue×5`) as a pre-pass to give the model a factual anchor.
- **Decomposer**: The two-phase clarity check (ask→decompose vs. decompose directly) was designed by me; Copilot generated the code once the branching logic was described.
- **Card Chat action dispatch**: Copilot generated the JSON action parser; I added the strict whitelist (`createSubtasks`, `setPriority` only) after realising the initial version would attempt to execute any action string the model returned.

#### Seed script
Prompted Copilot to write a seed script with: 2 users, 1 workspace, 1 board, 10 cards spread across 4 columns with mixed priorities, 2 overdue cards, and 1 seeded comment. It produced a working script in one pass; I adjusted the seeded due dates to ensure they were genuinely overdue relative to the test date.

---

### Honest assessment

Copilot was most useful for:
- **Eliminating boilerplate** — NestJS modules, Mongoose schemas, DTO classes, shadcn/ui compositions.
- **First working drafts** — even when the first version needed fixing, having something concrete to react to was faster than starting from scratch.
- **Remembering API shapes** — `@hello-pangea/dnd` callbacks, Mongoose operators, Next.js App Router conventions.

It was less reliable for:
- **State management across component boundaries** — Copilot treated each component in isolation; I had to design the data-flow topology (which state lives where) myself.
- **Security defaults** — CORS `*`, missing input whitelisting in the AI action dispatcher. Required a manual review pass.
- **Subtle Next.js caching** — `revalidatePath`, `Cache-Control`, server-only cookie forwarding needed manual correction every time.

## Overview

This section records the agentic behaviour implemented in `backend/src/ai/ai.service.ts` and the reasoning behind each design choice.

---

## Agents

### 1. Board Prioritizer (`prioritizeBoard`)

**Endpoint:** `POST /api/ai/boards/:boardId/prioritize`

**Trigger:** User clicks _AI Assist → Prioritize day_ in the board header.

**Multi-step process:**

| Step | What happens                                                                                                                                              |
| ---- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1    | Fetch full board data (columns + cards) via `CardsService.getBoardWithColumns`                                                                            |
| 2    | Compute a numeric urgency score per card: `priority_weight + age_days/7 + overdue×5` where `priority_weight` maps `high→3`, `medium→2`, `low→1`, `null→0` |
| 3    | Sort cards by score, assemble a JSON payload, call GPT with a system prompt asking for concise, action-oriented reasoning                                 |
| 4    | Return a ranked list `{ cards: [{ _id, title, urgencyScore, reasoning }], summary }`                                                                      |

**Prompt design:** The system prompt asks the model to act as a "senior project manager" and explain _why_ each card is important in ≤ 15 words. Using a small token budget keeps latency low and outputs scannable.

**Why a scoring pre-pass?** Seeding the LLM with computed scores gives it a factual anchor, reducing hallucinated priority judgments and making the ranking deterministic enough to trust.

---

### 2. Task Decomposer (`decomposeCard`)

**Endpoint:** `POST /api/ai/cards/:cardId/decompose`

**Trigger:** User clicks _Generate subtasks_ chip in the card AI chat, or the chat agent decides to decompose.

**Multi-step process:**

| Step | What happens                                                                                                                                     |
| ---- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| 1    | Fetch full card (title + description)                                                                                                            |
| 2    | Send to GPT with a _clarity check_ — does the card have enough detail to decompose?                                                              |
| 3a   | If not: return a `{ clarifyingQuestion }` response; the UI shows this as a chat message                                                          |
| 3b   | If yes: generate 2–5 concrete subtask titles                                                                                                     |
| 4    | If `createCards: true` in the request body, call `CardsService.createCard` for each subtask in the same column, then emit `broadcastBoardUpdate` |

**Design note:** The two-phase check (clarity → decompose) prevents generating vague sub-tasks like "Fix it" or "Look into this" when the parent card is itself under-specified. This is the core "agentic" loop — the agent can ask for more information rather than blindly executing.

---

### 3. Standup Generator (`generateStandup`)

**Endpoint:** `POST /api/ai/boards/:boardId/standup`

**Trigger:** User clicks _AI Assist → Generate standup_ in the board header.

**Multi-step process:**

| Step | What happens                                                                                                                           |
| ---- | -------------------------------------------------------------------------------------------------------------------------------------- |
| 1    | Fetch full board data                                                                                                                  |
| 2    | Categorize cards into `done` (last column), `inProgress` (middle columns), `blockers` (overdue or cards with "block"/"stuck" in title) |
| 3    | Prompt GPT with categorized lists; ask for a Slack-style async standup in three sections: ✅ Done, 🔄 In Progress, 🚨 Blockers         |
| 4    | Return `{ message, blockers: [{ title }] }`                                                                                            |

**Prompt design:** The system prompt explicitly tells the model to use emoji bullet points, keep the tone professional but human, and limit the message to ~200 words. The structured output format (three labelled sections) makes it copy-pasteable directly into Slack or Teams.

---

### 4. Card Chat (`cardChat`)

**Endpoint:** `POST /api/ai/cards/:cardId/chat`

**Trigger:** User types in the AI chat panel inside a card's detail modal.

**Architecture:**

The card's full context (title, description, column, assignees, priority, due date) is injected into the **system prompt** once per conversation. The user's message history is forwarded as `messages[]` to maintain context across turns.

The model is instructed that it _may_ respond with a JSON action block instead of plain text:

```json
{"action":"createSubtasks","subtasks":["Title 1","Title 2"]}
{"action":"setPriority","priority":"high"}
```

When the backend detects this pattern, it executes the action (creates cards / updates priority) and returns a confirmation message alongside the result. This lets the model drive mutations through natural conversation ("break this into subtasks" → model emits action → cards created automatically).

**Security note:** The JSON action parser uses a strict parsing step and only allows the two whitelisted action types. Arbitrary code execution is not possible.

## Model Configuration

| Setting       | Value                                                                     |
| ------------- | ------------------------------------------------------------------------- |
| Provider      | OpenAI                                                                    |
| Default model | `gpt-4o-mini`                                                             |
| Override      | `OPENAI_MODEL` env var                                                    |
| Temperature   | `0.3` (prioritizer, standup) / `0.5` (decomposer, chat)                   |
| Max tokens    | Not capped (relies on model default; responses are typically <500 tokens) |

`gpt-4o-mini` was chosen for the right balance of speed, cost, and reasoning quality for task-management payloads. All prompts are < 2 000 input tokens, keeping latency under 2 s in typical usage.

---

## Error Handling

- If `OPENAI_API_KEY` is missing, the service throws `InternalServerErrorException` with a clear message.
- If OpenAI returns a non-200, the NestJS service logs the error and surfaces `"AI service unavailable"` to the client.
- The frontend action wrappers (`src/features/ai/actions.ts`) always return `{ error?: string, result? }` — the UI shows the error inline without crashing.
